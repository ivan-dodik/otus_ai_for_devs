# Трудности при выполнении домашнего задания (опционально)

## 1. Выбор инструмента и изучение его документации

**Проблема:** Для Cline не существует единого официального документа "как подключить инструкции к проекту" — информация распределена по документации, репозиторию и community-дискуссиям.

**Решение:** Изучены:
- Репозиторий Cline на GitHub (раздел документации)
- Практика использования `.clinerules/` directory (подтверждено через community)
- Механизмы Memory Bank (описаны в `.clinerules/default-rules.md`)

## 2. Объём кодовой базы Betaflight

**Проблема:** Проект Betaflight содержит огромную кодовую базу с сотнями файлов, множеством модулей и сложной архитектурой. Создание полного контекста для ИИ потребовало значительного времени на анализ.

**Решение:** Сфокусировались на ключевых модулях, наиболее релевантных для типовых задач (RX, FC, Flight, Blackbox).

## 3. Проверка компиляции для blackbox-рефакторинга

**Проблема:** При рефакторинге blackbox потребовалось изменить `static` на `extern` для нескольких переменных, используемых как в init-, так и в runtime-коде. Это могло привести к ошибкам сборки на разных target-платформах.

**Решение:** Сборка выполнена для STM32F405 (один из основных target'ов). Для полной уверенности потребовалась бы сборка для всех поддерживаемых платформ, что заняло бы значительное время.

## 4. Unit-тесты: Google Test и stubs

**Проблема:** Для тестирования функций из `src/main/rx/rx.c` потребовалось создать stubs для внешних зависимостей (конфигурационные структуры, глобальные переменные, функции из других модулей). Некоторые функции имели сложные зависимости, что потребовало тщательного анализа.

**Решение:** Stubs созданы только для необходимых зависимостей. Использован существующий Makefile для тестов (`src/test/Makefile`), который уже содержит настройки для `rx_rx_unittest`.

## 5. Обеспечение следования инструкциям

**Проблема:** Требовалось убедиться, что AI-агент действительно следует созданным инструкциям, а не просто упоминает их.

**Решение:** Созданы проверочные требования (преамбула, обновление CHANGES/PROMPTS/PLAN), которые легко отследить. Проведены 3 практические проверки с документированием каждого критерия.

## 6. Синхронизация планов и документации

**Проблема:** При выполнении нескольких задач параллельно (debounce, blackbox, unit-тесты) требовалось синхронизировать обновления CHANGES.md, PROMPTS.md, PLAN.md и memory-bank.

**Решение:** Установлен строгий порядок: изменения кода → CHANGES.md → PROMPTS.md → PLAN.md → activeContext.md. Это обеспечило консистентность документации.
