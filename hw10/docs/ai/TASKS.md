# Типовые задачи

## Сценарий 1: Добавление поддержки ARM стиками как на DJI Mavic

**Описание:** Реализовать ARM/DISARM через комбинацию стиков, аналогично потребительским дронам DJI Mavic.

**Механика:**
- **ARM**: оба стика одновременно **вниз и вовнутрь** (левый стик — Throttle down + Yaw left, правый стик — Pitch down + Roll right). Удержание позиции **1 секунду**. Положение переключателя ARM switch **не должно учитываться** (приоритет стиков).
- **DISARM**: оба стика одновременно **вниз и наружу** (левый стик — Throttle down + Yaw right, правый стик — Pitch down + Roll left). При работающих моторах.
- Интегрировать с существующей ARM-логикой в `src/main/fc/`.

**Пример промпта:**
> "Реализуй ARM/DISARM через комбинацию стиков как на DJI Mavic: ARM — оба стика вниз и вовнутрь (Throttle down + Yaw left, Pitch down + Roll right) при удержании 1 сек. Положение ARM switch игнорировать. DISARM — оба стика вниз и наружу (Throttle down + Yaw right, Pitch down + Roll left) при работающих моторах."

**Ожидаемый формат ответа:**
1. План действий (какие файлы будут изменены, какие созданы)
2. Изменения в коде (полный diff или описание)
3. Пояснение решений (почему выбран такой подход)
4. Шаги проверки (как убедиться, что всё работает)

**Критерии готовности:**
- ARM работает через стики независимо от положения ARM switch
- DISARM — стики вниз и наружу при работающих моторах
- 1 секунда удержания для ARM + защита от случайного срабатывания
- Существующий механизм ARM/DISARM через переключатель не сломан
- Нет регрессий в существующих тестах
- Код соответствует `.clinerules/coding-style.md`

## Сценарий 2: Исправление бага при DISARM через переключатель (дебаунс)

**Описание:** При вибрациях дрона переключатель режимов генерирует ложные DISARM. Необходимо добавить программный антидребезг (debounce).

**Механика:**
- Состояние переключателя должно быть стабильным минимум **50 мс** перед изменением статуса ARM.
- Debounce реализовать программно (без изменения железа).

**Пример промпта:**
> "При вибрациях дрона переключатель режимов генерирует ложные DISARM. Добавь debounce-защиту: состояние переключателя должно быть стабильным минимум 50 мс перед изменением статуса ARM."

**Ожидаемый формат ответа:**
1. План действий
2. Изменения в коде
3. Пояснение решений
4. Шаги проверки

**Критерии готовности:**
- Ложные DISARM от вибраций устранены
- Debounce 50 мс (конфигурируемая константа)
- Задержка не мешает нормальному переключению (пилот не замечает задержки)
- Нет регрессий в существующей ARM-логике
- Код соответствует `.clinerules/coding-style.md`

## Сценарий 3: Рефакторинг модуля blackbox

**Описание:** Выделить логику инициализации из `src/main/blackbox/blackbox.c` в отдельный файл `blackbox_init.c`.

**Механика:**
- Сигнатуры функций не менять.
- `blackbox.c` должен только объявлять `extern`-функции инициализации.
- Все функции инициализации переместить в `blackbox_init.c`.

**Пример промпта:**
> "Вынеси логику инициализации blackbox из blackbox.c в новый файл blackbox_init.c. blackbox.c должен только объявлять extern-функции инициализации. Сигнатуры функций не менять."

**Ожидаемый формат ответа:**
1. План действий
2. Изменения в коде (создание `blackbox_init.c`, правки `blackbox.c` и `blackbox.h`)
3. Пояснение решений
4. Шаги проверки

**Критерии готовности:**
- Код компилируется (`make` без ошибок)
- Blackbox работает как раньше (нет изменений в поведении)
- Все функции инициализации находятся в `blackbox_init.c`
- Нет дублирования кода
- Код соответствует `.clinerules/coding-style.md`

## Сценарий 4: Добавление unit-тестов для модуля

**Описание:** Покрытие unit-тестами функции обработки RC каналов из `src/main/rx/`.

**Механика:**
- Выбрать одну или несколько функций из `src/main/rx/` (например, обработка маппинга каналов).
- Написать unit-тесты с использованием Google Test (см. `lib/test/gtest/`).
- Тесты должны покрывать: нормальные сценарии, граничные значения, ошибочные входы.

**Пример промпта:**
> "Напиши unit-тесты для функции обработки RC каналов из src/main/rx/. Используй Google Test. Покрой нормальные сценарии, граничные значения и ошибочные входы. Тесты должны компилироваться и проходить."

**Ожидаемый формат ответа:**
1. План действий (какие функции тестируются, какие файлы создаются)
2. Код тестов
3. Пояснение тестовых сценариев
4. Инструкция по запуску

**Критерии готовности:**
- Тесты компилируются и проходят
- Покрытие: минимум 3 тестовых случая (норма, граница, ошибка)
- Тесты используют Google Test (существующую инфраструктуру)
- Тесты не требуют железо (чистая логика)
- Код тестов соответствует стилю проекта
