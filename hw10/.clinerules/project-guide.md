# Betaflight Project Guide for AI Agents

Этот файл — основной набор инструкций для AI-агента при работе с проектом Betaflight. Все инструкции из этого файла, а также из `.clinerules/coding-style.md`, `.clinerules/default-rules.md` и `.clinerules/update_prompts.md`, ДОЛЖНЫ соблюдаться автоматически.

## 1. Контекст и границы

### Назначение проекта

Betaflight — прошивка для полётных контроллеров (FC) FPV-дронов и других мультироторных летательных аппаратов с открытым исходным кодом. Проект обеспечивает:

- Стабилизацию и управление полётом (PID-регуляторы)
- Обработку сигналов с пульта дистанционного управления (RC)
- Управление моторами (DSHOT, Oneshot, PWM)
- Телеметрию, OSD (экранная индикация), чёрный ящик (blackbox)
- Настройку через CLI, MSP, CMS

### Ограничения

**Что НЕЛЬЗЯ менять без явного разрешения:**

- Драйверы устройств в `src/main/drivers/` без необходимости
- Публичные API модулей (заголовочные файлы, используемые из других модулей)
- Конфигурационные структуры в `src/main/pg/` без миграции
- Target-специфичный код за пределами `src/main/target/`
- Протоколы коммуникации (MSP, CRSF, MAVLink и т.д.)

**Архитектурные ограничения:**
- No "upwards" calls: драйверы (нижний слой) не должны вызывать код из верхних слоёв (FC, flight, io)
- No target-specific code вне `src/main/target/`
- Максимальная длина строки: 120 колонок (140 — жёсткий лимит)

### Правила взаимодействия с агентом

- **Уточнять, если задача неоднозначна.** Если промпт допускает множественные трактовки — задать уточняющий вопрос.
- **Не делать допущений о железе.** Если задача требует знаний о конкретной плате — уточнить.
- **Перед внесением изменений составить план.** Никаких изменений без предварительного плана.

## 2. Требования к результату

### Формат ответа AI-агента

Каждый ответ, содержащий изменения в коде, ДОЛЖЕН следовать структуре:

1. **План действий** — какие файлы будут изменены/созданы/удалены, в какой последовательности
2. **Изменения в коде** — полный diff или содержимое новых/изменённых файлов
3. **Пояснение решений** — почему выбран такой подход, какие альтернативы рассматривались
4. **Шаги проверки** — как убедиться, что изменения корректны

### Стиль кода

Весь новый код и изменения ДОЛЖНЫ соответствовать `.clinerules/coding-style.md`. Ключевые требования:
- 1TBS (K&R variant), 4 пробела, без табов
- Braces обязательны для всех if/else (даже для одной строки)
- lowerCamelCase для функций и переменных
- Float-константы с суффиксом `f`: `1.0f`
- `#pragma once` для заголовочных файлов
- Prefer `const` over pre-processor macros
- No continue/goto/multiple return без веской причины

### Definition of Done

Перед завершением задачи агент ДОЛЖЕН проверить все пункты из `docs/ai/DEFINITION_OF_DONE.md`.

## 4. Проверочные требования

### Преамбула

Каждый новый `.c` / `.h` файл ДОЛЖЕН содержать преамбулу — комментарий в начале файла с описанием его назначения. Пример:

```c
/* Этот файл содержит реализацию ARM/DISARM через комбинацию стиков.
 * Реализует механику как на DJI Mavic: оба стика вниз и вовнутрь для ARM,
 * оба стика вниз и наружу для DISARM.
 */
```

### Документирование изменений

После каждого изменения (создания или модификации файлов) агент ДОЛЖЕН:

1. **Обновить `CHANGES.md`** — добавить запись с датой, описанием изменений и списком затронутых файлов
2. **Обновить `PROMPTS.md`** — добавить запись о промпте, который привёл к изменениям
3. **Обновить `docs/ai/PLAN.md`** — отметить галочками выполненный прогресс
4. **Обновить `memory-bank/activeContext.md`** — если изменения существенные

### Качество кода

- Все изменения ДОЛЖНЫ соответствовать `.clinerules/coding-style.md`
- Нет неиспользуемых `#include`
- Нет закомментированного кода (если он не является частью документации)
- Нет `TODO`/`FIXME`/`HACK` в новом коде без объяснения причины
- Новые конфигурационные параметры добавляются в `src/main/pg/`

## 5. Проверка следования инструкциям

Для проверки, что агент действительно следует инструкциям, используйте:

1. **Тест преамбулы** — попросите создать новый `.c` файл и проверьте наличие преамбулы
2. **Тест обновления CHANGES** — попросите внести любое изменение и проверьте `CHANGES.md`
3. **Тест формата ответа** — попросите решить задачу и проверьте структуру: план → изменения → пояснение → проверка
4. **Тест coding-style** — попросите написать функцию и проверьте соответствие стилю (4 пробела, braces, naming)
