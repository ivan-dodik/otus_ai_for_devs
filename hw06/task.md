# Pевью кода opensource проекта

## Цель

* научиться использовать AI-ассистента для анализа чужого кода и проверки своих выводов;
* научиться быстро ориентироваться в незнакомом репозитории: понимать структуру, точки входа, назначение модулей;
* потренироваться находить реальные проблемы в коде: потенциальные баги, некорректные решения, архитектурные и читабельностные изъяны;
* научиться формулировать замечания так, чтобы их понял разработчик, не знакомый с технологией проекта;
* оформить обзор, близкий к формату профессионального code review.

## Требования к выбору проекта

Любой публичный репозиторий на GitHub, который удовлетворяет условиям:

**1. Вы знаете язык / платформу проекта.**<br>Вам должно быть комфортно читать код без изучения языка «с нуля».

**2. Проект не слишком прост.**<br>Подходящий проект должен содержать:

* несколько модулей / пакетов;
* функции или классы средней сложности;
* реальную логику (API, обработка данных, небольшая библиотека, CLI, сервис);
* историю коммитов и структуру, позволяющую провести анализ.<br>
Не подходят: однофайловые учебные примеры, «Hello world», проекты до 20–30 строк.

**3. Адекватный объём.**<br>Не берите огромный фреймворк. Если проект крупный — выберите один его пакет или директорию.

## Требования к оформлению ревью

Ваше ревью должно быть:

**1. Конкретным.**<br>Каждое замечание должно включать:

* название файла;
* ссылку или указание номера строки;
* небольшой фрагмент кода (при необходимости);
* объяснение, почему это проблема.

**2. Понятным проверяющему**<br>Проверяющий — опытный инженер, однако он может не знать вашу платформу или библиотеку.

Поэтому формулируйте:

* чётко и без жаргона;
* с минимально необходимыми пояснениями контекста;
так, чтобы смысл был очевиден даже без глубокого знания технологии.

**3. Структурированным**<br>Рекомендуемая структура:

* краткое описание проекта и выбранной области анализа;
* найденные проблемы, сгруппированные по типам (баги, архитектура, читаемость, тесты, документация);
* конкретные примеры кода;
* предложения улучшений;
* итоговый вывод: текущее состояние проекта и основные приоритеты.

**4. С приложением всех промптов**<br>Все запросы к AI, использованные в процессе анализа, обязательно сохраните и приложите списком в конце работы.

**5. Формат сдачи**<br>Скопируйте себе документ по ссылке и заполните его.


## Критерии оценки

Работа считается выполненной, если:

* проект выбран корректно и соответствует требованиям;
* ревью содержит понятные и конкретные замечания с ссылками на код;
* формулировки ясны человеку, который не знаком с технологией проекта;
* предложены разумные рекомендации по улучшениям;
* приложен полный список использованных промптов.

## Компетенции

* сопровождать и оптимизировать проекты с помощью AI
* использовать AI для навигации по большим кодовым базам
* вести обсуждения архитектурных вариантов с AI
* оценивать риски и альтернативы
