# Политика безопасности

## 🔒 Обзор безопасности

Приложение "Мини-анкета" реализует следующие меры безопасности для защиты данных пользователей и предотвращения злонамеренного использования.

## 🛡️ Реализованные меры безопасности

### 1. CORS (Cross-Origin Resource Sharing)

**Настройка:**
- Разрешены только определенные домены:
  - `http://localhost:8080` (frontend)
  - `http://localhost:5000` (backend)
  - `http://127.0.0.1:8080`
  - `http://127.0.0.1:5000`

**Цель:** Предотвращение межсайтовых запросов с нежелательных доменов.

### 2. Валидация входных data

**Проверки:**
- Все обязательные поля должны присутствовать
- Типы данных должны соответствовать ожиданиям (строка/число)
- Длина строковых полей ограничена 500 символами
- Возрастные значения должны быть между 0 и 150
- Опыт работы должен быть между 0 и 100
- Поля не могут быть пустыми

**Цель:** Предотвращение инъекций, XSS атак и некорректных данных.

### 3. Ограничение скорости запросов (Rate Limiting)

**Настройка:**
- По умолчанию: 200 запросов в день, 50 запросов в час
- Для отправки ответов: 10 запросов в минуту

**Цель:** Предотвращение DDoS атак и злоупотребления API.

### 4. Обработка ошибок

**Принципы:**
- Сообщения об ошибках не раскрывают внутреннюю структуру
- Stack traces не отправляются клиенту
- Логируются только необходимые данные

**Цель:** Предотвращение утечки конфиденциальной информации.

## ⚠️ Известные ограничения

### 1. Хранение данных

**Текущая реализация:**
- Данные хранятся только в памяти приложения
- Данные теряются при перезапуске сервера
- Нет шифрования данных at rest

**Рекомендации для production:**
- Использовать базу данных с шифрованием
- Реализовать резервное копирование
- Настроить политики хранения data

### 2. Аутентификация и авторизация

**Текущая реализация:**
- Нет аутентификации пользователей
- Нет разграничения прав доступа
- Любой может просмотреть все ответы

**Рекомендации для production:**
- Добавить аутентификацию (JWT, OAuth)
- Реализовать роли пользователей
- Ограничить доступ к административным функциям

### 3. HTTPS

**Текущая реализация:**
- Приложение работает по HTTP
- Данные передаются в открытом виде

**Рекомендации для production:**
- Настроить HTTPS с SSL/TLS сертификатами
- Принудительно перенаправлять HTTP на HTTPS
- Использовать secure cookies

### 4. CSRF защита

**Текущая реализация:**
- Нет явной CSRF защиты
- CORS частично защищает от межсайтовых запросов

**Рекомендации для production:**
- Добавить CSRF токены
- Использовать SameSite cookies
- Реализовать дополнительную проверку origin

## 🚨 Рекомендации по развертыванию

### Для локальной разработки:
1. Запускайте приложение только на localhost
2. Не открывайте порты во внешнюю сеть
3. Используйте брандмауэр для ограничения доступа

### Для production:
1. **Никогда** не используйте debug режим
2. Настройте HTTPS с валидным SSL сертификатом
3. Используйте reverse proxy (nginx, Apache)
4. Настройте мониторинг и логирование
5. Регулярно обновляйте зависимости
6. Проводите security аудит кода
7. Используйте WAF (Web Application Firewall)

## 📊 Классификация данных

**Текущие данные:**
- Имя пользователя (текст)
- Возраст (число)
- Любимый язык программирования (текст)
- Опыт работы (число)
- Предпочтительная среда разработки (текст)
- Временная метка отправки

**Категория:** Низкая чувствительность (не персональные данные в соответствии с GDPR)

**Требования к хранению:**
- Не требуется специальное шифрование
- Рекомендуется анонимизация при длительном хранении
- Следует установить политику удаления data

### Заголовки безопасности:

**Рекомендуемые заголовки:**
```
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 1; mode=block
Content-Security-Policy: default-src 'self'
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

## 📝 Процедуры безопасности

### При обнаружении уязвимости:
1. Немедленно задокументировать уязвимость
2. Оценить уровень риска (низкий/средний/высокий/критический)
3. Разработать и протестировать исправление
4. Применить исправление в production
5. Провести пост-мортем анализ

### Регулярные задачи:
- Еженедельно: проверка логов на подозрительную активность
- Ежемесячно: обновление зависимостей
- Ежеквартально: security аудит кода
- При изменении: тестирование безопасности
